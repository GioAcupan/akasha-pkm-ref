Canvas Generation: How it works

The Vision LLM produces valid JSON Canvas 1.0 from a photo of a diagram

Prompt structure

The LLM receives the image with instructions to produce JSON Canvas output:

"This is a photo of a hand-drawn diagram. Transcribe it to JSON Canvas 1.0 format.
Rules:
• Each bubble/box → text node with transcribed handwriting
• Arrows/lines → edges with directional arrows
• Labels on arrows → edge.label text
• Group related nodes with group nodes (labeled rectangles)
• Use colors: green for correct/yes, red for wrong/no, blue for neutral
• Position nodes proportionally to their layout on the page
• The canvas file will be embedded in a note — text nodes only, no file/image nodes needed here
• Output ONLY valid JSON Canvas — no markdown wrapping, no explanation"

Validation

✅ Must check
  • Valid JSON parse
  • Has "nodes" array
  • Each node has id, type, text, x, y
  • 16-char hex IDs per spec
  • Edge references are valid node IDs
  • No prose wrapper, no ```json
⚠️ Confidence check
  • Node count < 3 → low confidence (too sparse)
  • All node text is empty → low confidence
  • Many "[unreadable]" entries → flag
  • Coordinates don't match spatial layout → low confidence
  • Low confidence → fall back to image-link note

Canvas file storage

Canvas files are first-class knowledge artifacts. They live alongside notes:

Knowledge/cs/
├── binary-search-trees.md
├── binary-search-trees.canvas  <-- paired artifact
├── sorting-algorithms.md
├── sorting-algorithms.canvas  <-- paired artifact
└── _moc-registry.md

The .md file embeds the .canvas with ![[filename.canvas]]. Canvas files are NOT listed in _moc-registry — the .md note carries all the frontmatter and navigation metadata.
