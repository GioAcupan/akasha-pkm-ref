Proposed: akasha-diagram-parser skill

A dedicated skill the mobile parser invokes when it encounters diagram content

Why a skill?

The mobile parser agent currently has one job: "transcribe this math photo to LaTeX." Adding diagram parsing inline makes it bloated. A skill keeps the parser focused on routing — "is this math, diagram, or sketch?" — and delegates the format details to a specialized skill that can be tested and refined independently.

Structure (references/ pattern)

.commandcode/skills/akasha-diagram-parser/
├── SKILL.md
│   Main instruction: classification rules,
│   fallback chain, output specifications
├── references/
│   ├── canvas-spec.md
│   │   JSON Canvas 1.0 spec (from obsidian-skills)
│   ├── canvas-examples.md
│   │   Mind map, system diagram examples
│   ├── mermaid-flowchart.md
│   │   Flowchart syntax for algorithm/process diagrams
│   ├── mermaid-mindmap.md
│   │   Mind map syntax (for simple hierachical ones)
│   ├── mermaid-sequence.md
│   │   Sequence diagram syntax
│   ├── validation-rules.md
│   │   What counts as pass/fail for each format
│   └── failures.md
│       Common failure modes and how to detect them
└── (adapts from existing skills, not duplicating)

What goes in each file

SKILL.md
  • Purpose: "Parse handwritten diagram photos into structured outputs"
  • Classification rubric (math vs diagram type)
  • Workflow: classify → select format → generate → validate → fallback if needed
  • Output spec: when to produce .canvas, when to produce Mermaid block, when to produce image-link note
  • Which reference doc to read for each diagram type
References
  • From obsidian-skills/json-canvas: Node/edge spec, ID generation, layout guidelines, validation checklist
  • From mermaid-skill: Flowchart, mindmap, sequence diagram syntax docs
  • New: Validation rules specific to transcription quality (confidence thresholds, [?] detection, node count minimums)
  • New: Failure detection (common garbled output patterns)
