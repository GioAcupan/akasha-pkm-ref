Impact on the mobile capture contract

The manifest.json contract between the app and the parser — does it change?

Current manifest.json

{
"domain": "cs",
"mocs": ["dsa", "trees"],
"timestamp": "2026-07-01T08:00:00Z",
"images": ["page-01.jpg", "page-02.jpg"]
}

Does the app need to tag image types?

Two options for the manifest contract:

A

No contract change — parser handles classification

Manifest stays as-is. The parser's Vision LLM classifies each image independently (math vs diagram vs sketch). The domain field is used as a hint but the parser can redirect per-image. Works because the classifier is built into the same Vision call that does transcription.

Pro

  • Zero app-side changes needed
  • Classification is nearly free (same Vision call)

Con

  • No human override — if the LLM misclassifies, it's wrong
B

Add optional type field to manifest

Add a type field per image in the manifest. User selects "math", "diagram", or "mixed" per photo when scanning. The parser uses this as a strong hint.

"images": [
{"file": "page-01.jpg", "type": "math"},
{"file": "page-02.jpg", "type": "diagram"}
]

Pro

  • User can correct misclassification at capture time

Con

  • Requires mobile app changes
  • Extra tap per photo during capture

My recommendation: start with A

The Vision LLM is reliable enough at classifying page content (it can see the whole image). Starting with no contract change means the mobile app doesn't block the diagram feature. If we find misclassifications in practice, we can add the type hint later — the parser can be updated to read it without breaking the existing manifest format.
